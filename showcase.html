Skills — Workflows in Plain English

+
homelab-add-service "add SABnzbd to the homelab"

Writes compose entry, env vars, deploys to NAS01, adds NPM proxy host + Uptime Kuma monitor. Handles port conflict checking and appdata dir creation.

A
homelab-service-audit "audit my Tdarr config"

Checks against official docs + community best practices. Prioritised Hi/Med/Lo findings with specific remediation steps.

H
homelab-health-check "how's the stack looking?"

Parallel health check across all services: Docker containers, TrueNAS pool, iDRAC sensors, Duplicati last run. Produces a ✓/⚠/✗ summary report.

U
homelab-update-review "what updated last night?"

Reads Watchtower logs, checks health of every updated container, flags regressions. Runs every morning after the 4 AM Watchtower cron.

G
homelab-git-push "save my changes"

Checks what changed, sweeps docs for anything stale, writes a descriptive commit message, stages, commits, and pushes to GitHub. One command.

Case Study: Debug

Real Example: "Why Did 1,003 Files Error?"

1

The symptom

1,003 files batch-stamped "Transcode error" overnight. No obvious pattern. Workers showing idle despite 7,000+ files queued.

2

Claude digs in

Reads Watchtower logs → spots NAS02-node restart timestamp → cross-references tdarr server update time. Timestamps align exactly.

3

Root cause

enableDockerAutoUpdater=true caused the node to restart mid-transcode when Watchtower updated the server image. All in-flight transcodes errored.

4

Fix + prevention

Pin both images to 2.80.01. Add watchtower.enable=false label. Set enableDockerAutoUpdater=false on NAS02-node.

5

Recovery

Live Python script via docker exec updates 1,003 SQLite rows from "Transcode error" → "Queued" — no container restart required. All files reprocessed overnight.

Total debugging time: ~15 minutes. Without CLAUDE.md context: hours of log archaeology.
