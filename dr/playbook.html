🔄 Restore Procedure
Click each step to mark it complete. Times are estimates for a warm machine (Docker images cached). Add ~15-20 min for a cold machine.
🔀 Which restore path?
Path A — NAS02 available
NAS01 dead, NAS02 fine. Local restore from NAS backups share.
RTO: ~30 min
Run: ./restore.sh
Path B — NAS02 also gone
Both dead, or restoring off-site. Downloads ~46 GB from B2. Media re-downloads via Radarr/Sonarr after.
RTO: ~60-90 min
Run: ./restore.sh --source b2
Phase 1
Before You Touch Anything
~0 min
1
Confirm the NAS01 is actually dead
Verify this isn't a temporary glitch. Try the NAS01 admin UI, ping 192.168.1.33, check physical LEDs. If it's just a hung container, reboot the NAS01 via power cycle before escalating to full DR.
2
Check NAS02 status — determines which path to take
If NAS02 is up → Path A (local restore, 30 min). If NAS02 is also down or unreachable → Path B (B2 offsite restore, 60-90 min, media re-downloads afterwards).
ping 192.168.1.31 # NAS02 reachable? showmount -e 192.168.1.31 # NFS exports visible?
💡 If NAS02 is reachable, proceed with Path A. If not, proceed with Path B — the restore script handles all the differences automatically via --source b2.
3
Locate your secrets file
You need ~/.homelab.secrets (or a copy from your password manager / USB key). Path A needs NAS02 share paths + Duplicati passphrase. Path B additionally needs B2_KEY_ID and B2_APPLICATION_KEY.
⚠ Without this file the restore script will stop at step 1. Have it ready before proceeding.
Phase 2
Set Up the DR Host
~5 min
4
Provision a Linux host
Any x86_64 Linux box with internet access and ~50GB of free disk (for Docker images and appdata). Ubuntu 22.04 LTS is the reference. The P360 running Ubuntu or a VM with 4+ cores and 8GB RAM is fine.
💡 Docker images alone take ~15 GB. The Plex DB and configs will add several more GB.
5
Copy secrets file and SSH key onto the DR host
From your password manager or USB key, place the secrets file and GitHub SSH key on the DR host.
# Secrets file cp /path/to/.homelab.secrets ~/.homelab.secrets chmod 600 ~/.homelab.secrets # GitHub SSH key (needed to clone the repo) cp /path/to/id_ed25519 ~/.ssh/id_ed25519 chmod 600 ~/.ssh/id_ed25519 ssh-add ~/.ssh/id_ed25519 # Test GitHub access ssh -T git@github.com
💡 Store your SSH private key in your password manager (1Password, Bitwarden) alongside the secrets file so both are available during DR.
Phase 3
Run the Restore Script
~25 min
6
Clone the homelab repo and run restore.sh
The repo is private — SSH auth is required. The SSH key you placed in step 5 handles this. The script handles everything: Docker install, NFS mounts, Duplicati restore, .env generation, and container startup.
# Path A — NAS02 available (~30 min) git clone git@github.com:yourusername/homelab.git ~/homelab chmod +x ~/homelab/dr/restore.sh ~/homelab/dr/restore.sh # Path B — B2 offsite (~60-90 min, downloads ~46 GB) git clone git@github.com:yourusername/homelab.git ~/homelab chmod +x ~/homelab/dr/restore.sh ~/homelab/dr/restore.sh --source b2
💡 The script is idempotent — safe to re-run if it fails partway through. It also starts its own ssh-agent internally so it can re-clone the repo for compose files.
⚠ The script detects if your user isn't in the docker group yet (common on fresh Ubuntu) and falls back to sudo docker automatically — no action needed.
7
If Duplicati CLI restore fails: use the web UI fallback
Start a standalone Duplicati container, open the UI, and run a restore manually.
Path A (NAS02) fallback
docker run -d --name dup-restore -p 8200:8200 \ -v /mnt/homelab/backups:/backups \ -v /opt/homelab/appdata:/restore \ lscr.io/linuxserver/duplicati:latest # Open http://localhost:8200 # Add Backup → Import existing → file:///backups/homelab-appdata # Restore → choose latest version → target /restore
Path B (B2) fallback
docker run -d --name dup-restore -p 8200:8200 \ -v /opt/homelab/appdata:/restore \ lscr.io/linuxserver/duplicati:latest # Open http://localhost:8200 # Add Backup → Backblaze B2 # Bucket: yourusername-appdata-backup # Key ID: (from ~/.homelab.secrets B2_KEY_ID) # Application key: (from ~/.homelab.secrets B2_APPLICATION_KEY) # Passphrase: (from ~/.homelab.secrets DUPLICATI_PASSPHRASE) # Restore → choose latest version → target /restore
7b
Path B only: trigger media re-download via Radarr & Sonarr
After the B2 restore, Radarr and Sonarr will see your full library but all files will be missing (NAS02 was unavailable). Once NAS02 is recovered and NFS mounts are up, trigger re-indexing. Or if NAS02 is permanently gone, trigger re-downloads from Usenet.
# After NAS02 is back and NFS mounts are restored: # Radarr → Movies → Wanted → Search All Missing # Sonarr → Series → Wanted → Search All Missing # SABnzbd will queue downloads — ~100 MB/s via Usenet # At that rate, re-downloading all media takes time but is automatic
💡 If this is a Path A restore, skip this step — media is already available via NFS from the NAS02.
Phase 4
Plex Reclaim
~3 min
8
Get a fresh Plex claim token
Plex needs to be claimed on the new host. The token expires in 4 minutes — have your terminal ready before visiting the link.
# 1. Visit https://www.plex.tv/claim and copy the token # 2. Edit .env: nano ~/homelab/compose/.env # Set: PLEX_CLAIM=claim-XXXXXXXXXX # 3. Recreate plex container to pick up the claim: cd ~/homelab/compose docker compose up -d plex
⚠ Use "up -d" not "restart" — restart doesn't re-read env vars.
9
Verify library path and clear the claim token
Open Plex, check that libraries show up. If paths are wrong, edit each library root to /media/movies, /media/tv, /media/4k, /media/music. Then clear the claim token.
# After Plex is claimed and libraries look good: # Edit .env → set PLEX_CLAIM= docker compose up -d plex # restart to clear claim env var
Phase 5
Reconnect Services
~5 min
10
Update Prowlarr app URLs
Prowlarr stores the IP of each arr app. Container names resolve internally, so if everything is on the same Docker network this may already work. If not, update each app in Prowlarr → Settings → Apps.
# Internal container URLs (preferred — use container name): # Sonarr: http://sonarr:8989 # Radarr: http://radarr:7878 # Radarr4K: http://radarr4k:7878 ← internal port, NOT 7070 # Bazarr: http://bazarr:6767
11
Verify TransmissionVPN tunnel
Confirm the VPN is routing traffic, not exposing your home IP.
docker exec transmissionvpn curl -s ifconfig.me # Should return a PIA IP — NOT your home IP (YOUR_PUBLIC_IP)
💡 TUN device: the restore script runs sudo modprobe tun but on modern kernels (including Ubuntu VMs) TUN is often built in — lsmod | grep tun returning nothing is normal if /dev/net/tun exists and TransmissionVPN starts correctly.
12
Update router port forward
In UniFi, update the port forward rule for 32400 → new DR host IP. Plex remote access depends on this. Also update the 80/443 port forwards to point at the DR host for NPM.
# UniFi OS → Firewall & Security → Port Forwarding # Rule: Forward TCP 32400 → [DR host IP]:32400 # Rule: Forward TCP 80 → [DR host IP]:80 (NPM HTTP / Let's Encrypt) # Rule: Forward TCP 443 → [DR host IP]:443 (NPM HTTPS)
13
Reconfigure Nginx Proxy Manager
NPM's proxy host config is stored in appdata and restored by Duplicati — it should come up automatically with all proxy hosts intact. Verify at http://[DR host]:81. If the wildcard cert needs renewal, NPM will handle it automatically via Cloudflare DNS challenge using the key in your .env.
💡 If proxy hosts are missing, re-add them manually: plex.yourdomain.com → plex:32400, seerr.yourdomain.com → seerr:5055. Select the *.yourdomain.com wildcard cert for both.
⚠ QTS runs on port 8443 on the NAS01 — if restoring to the NAS01 itself, ensure port 443 is free for NPM before starting it.
14
Trigger Recyclarr sync
Recyclarr runs daily automatically, but trigger a manual sync after DR to ensure quality profiles are current.
docker exec recyclarr recyclarr sync
✅ Post-Restore Verification
✓

cd ~/homelab/compose && docker compose ps
✓
Plex plays a file (internal)
Open Plex, play 30 seconds of a movie. Confirms NFS media mount and library paths are correct.
✓
Plex remote access enabled
Settings → Remote Access → should show "Fully accessible outside your network"
✓
Sonarr & Radarr health checks clear
System → Status in each arr app — should show green (or only minor indexer warnings).
✓
VPN IP confirmed ≠ home IP
docker exec transmissionvpn curl -s ifconfig.me # Expected: PIA IP, not YOUR_PUBLIC_IP
✓
Duplicati backup job scheduled on new host
Import ~/homelab/duplicati/homelab-appdata-backup.json into Duplicati at :8200 (Add Backup → Import from file). Run a manual backup to confirm it works from the new location.
✓
Uptime Kuma monitors are green
Open :3001 — all monitors should go green once containers stabilize. Update any monitors that reference the old host IP.

🔧 DR Maintenance Tasks
Keep these current so the 30-min target stays achievable.
Monthly
Test restore on DR host
Run restore.sh to a VM and verify containers come up and Plex plays. The only way to know your backup works is to test it.
Monthly
Pre-pull Docker images on DR host
Eliminates the 15-20 min image pull from the RTO. Run on whatever machine you'd use for DR.
cd ~/homelab/compose && docker compose pull
Monthly
Verify both Duplicati jobs in UI
Open :8200, check both "Homelab AppData → NAS02" and "Homelab AppData → B2" last run status. Run "Verify files" on B2 job periodically to confirm offsite integrity.
On change
Update B2_KEY_ID / B2_APPLICATION_KEY in password manager
If you rotate B2 application keys, update ~/.homelab.secrets and re-sync to your password manager. The B2 restore path requires these credentials.
On change
Update ~/.homelab.secrets in password manager
After any credential change (PIA password rotation, new Duplicati passphrase), update the secrets file and re-sync to your password manager.
On change
Commit compose changes to GitHub
The repo is the source of truth for the stack definition. Any time you add/remove/change a service, commit it.
cd ~/homelab && git add compose/ && git commit -m "describe change" && git push
